# Purpose

Enable recruiters to import candidate Resume documents into the Workspace.

Resume Upload is the primary entry point for Candidate Acquisition.

Every uploaded Resume creates an immutable Resume version and triggers the parsing pipeline.

This workflow does not create Candidates directly — it initiates the event chain that may lead to Candidate creation.

---

# Trigger

Recruiter uploads a Resume file via:

- Drag-and-drop in Candidate workspace
- File picker in Import dialog
- Bulk upload (multiple files — each file triggers separate WF-001 instance)
- API: `POST /candidates/import-resume` (Public Language)

---

# Preconditions

- Actor is authenticated Recruiter with `resume:upload` permission
- Actor belongs to active Workspace
- File format is supported: PDF, DOCX, DOC, PNG, JPG (scanned)
- File size ≤ 10 MB per file
- Workspace storage quota not exceeded

# Input

**Public API request** (`POST /candidates/import-resume`):

```json
{
  "file": "<binary>",
  "filename": "john_doe_resume.pdf",
  "source_type": "manual_upload",
  "candidate_id": null,
  "tags": [],
  "notes": null
}
```

**Internal command** (after auth — not exposed to client):

| Field | Source |
|-------|--------|
| workspace_id | auth context |
| actor_id | auth context |
| correlation_id | generated by CandidateImportService |
| mime_type, file_size_bytes | detected server-side |

| Field | Validation |
|-------|------------|
| file | Required, non-empty |
| filename | Required, max 255 chars |
| mime_type | Must be in allowed list |
| file_size_bytes | ≤ 10,485,760 |
| candidate_id | Optional — if provided, links Resume to existing Candidate |
| source_type | Enum: manual_upload, email_import, bulk_import |

---

# Output

**Public API response** (no internal event names):

```json
{
  "request_id": "req_xxx",
  "resume_id": "resume_xxx",
  "status": "processing",
  "message": "Resume imported successfully. Parsing in progress."
}
```

**Internal** (after EV-002): resume_id, resume_version, checksum — consumed by event handlers only.

# State Transition

| Entity | From | To | Condition |
|--------|------|-----|-----------|
| Resume | — | Created | Always on successful store |
| Candidate | — | New (pending) | Only if new Candidate path — via downstream WF-002 |

Candidate lifecycle state `Imported` is set by WF-002/WF-004, not WF-001.

# Event Sourcing View

```
Recruiter uploads file
        ↓
ResumeUploaded          EV-001
        ↓
ResumeStored            EV-002
        ↓
ResumeParsingRequested  EV-003  (triggers WF-004)
        ↓
[WF-004 Resume Parsing continues chain]
        ↓
ResumeParsed            EV-004
        ↓
CandidateCreated        EV-005  (if new)
        ↓
EmbeddingGenerated      EV-010
        ↓
KnowledgeUpdated        EV-011
        ↓
TimelineEntryCreated    EV-025
        ↓
NotificationCreated     (system)
```

WF-001 owns EV-001 and EV-002. EV-003 is emitted at handoff to WF-004.

---

# Validation Rules

From `04-business-rules.md`:

- Resume is immutable once stored (EV-002)
- Resume always belongs to exactly one Candidate (enforced when Candidate linked)
- Resume parsing never modifies original files
- Every upload creates new Resume Version
- Workspace isolation — no cross-workspace upload
- Source is permanent metadata

---

# Error Cases

| Error Code | Cause | Response | Event Emitted |
|------------|-------|----------|---------------|
| `FILE_TOO_LARGE` | file > 10 MB | Reject, user message | None |
| `UNSUPPORTED_FORMAT` | mime_type not allowed | Reject, list supported formats | None |
| `STORAGE_QUOTA_EXCEEDED` | Workspace limit reached | Reject, contact admin | None |
| `STORAGE_FAILURE` | Blob store unavailable | Retry 3x, then fail | None |
| `INVALID_CANDIDATE_ID` | candidate_id not in Workspace | Reject | None |
| `PERMISSION_DENIED` | Actor lacks permission | 403 | None |
| `CHECKSUM_MISMATCH` | Storage corruption detected | Fail, alert admin | None |

Failed uploads do not emit business events. Technical errors logged to Audit Log only.

# API Contracts

> **Public Language** — business-meaningful API. Events are internal only.

## Import Resume

```http
POST /api/v1/candidates/import-resume
```

Handled by `CandidateImportService.importResume()`.

See `12-application-services/CandidateImportService.md` for full contract.

**Response 202:** `{ request_id, resume_id, status, message }` — no event names exposed.

## Get Import Status

```http
GET /api/v1/candidates/import-resume/{resume_id}/status
```

Returns: `status: processing | parsed | failed`, optional `candidate_id`.

# Common Mistakes

❌ `POST /events/ResumeUploaded` — events are internal, not public API

❌ Exposing event names in API response

❌ Modifying stored Resume blob after upload

❌ Skipping EV-001/EV-002 — writing directly to parsing table

❌ Creating Candidate in WF-001 — defer to WF-002/WF-004

❌ Omitting `source_workflow: WF-001` from Audit

❌ Allowing Client persona to upload

❌ Cross-workspace file storage

❌ Deleting Resume on parse failure

❌ Synchronous parsing in upload request — must async via EV-003

❌ Treating upload as CRUD update
